# 위키 정리 설계안 — 2026-05-04

## 목표

위키 전체를 훑어 **AI 에이전트가 사용자(worldcar265)의 페르소나를 학습하기 좋은 형태**로 정돈한다. 동시에 위생 점검(깨진 링크·빈 노트·중복)도 같이 처리.

**Why**:
- 향후 Claude Project 또는 SDK 기반 페르소나 에이전트 구축 시, 위키가 1차 학습 자료가 됨.
- 노트가 60+개로 늘어나면서 자연스럽게 발생하는 중복·고립을 한 번 정돈할 시점.

**왜 지금**:
- 사용자가 GitHub 동기화·백업 셋업을 끝낸 뒤(2026-05-01)이고, 깃 안전망이 작동함.
- 정돈된 상태에서 출발하면 이후 노트 작성 흐름이 깔끔해짐.

## 결정 사항 (사용자와 합의)

| 항목 | 선택 |
|---|---|
| 정리 목적 | **페르소나 학습 1순위** + 위생 점검 동반 |
| 수정 적극성 | **다 알아서** (깃 + 별도 브랜치 + 아카이브 폴더가 안전망) |
| 정리 깊이 | **중간** — 기존 폴더 구조 존중, 새 메타 노트 신설 위주 |
| 중복 통합 기준 | **균형** — 같은 주제 + 70% 이상 겹칠 때 통합. 결정 근거를 커밋 메시지에 명시 |

## 안전장치

1. 작업 브랜치: `cleanup/2026-05-04` (main은 그대로)
2. 통합으로 사라지는 원본은 삭제 대신 `위키/_아카이브/`로 이동
3. 커밋을 단계별로 분리 → 부분 되돌리기 가능
4. 작업 완료 후 사용자가 GitHub에서 main 대비 변경사항 검토 후 머지 결정

## 7단계 + 7커밋

### 1단계 — 점검 리포트 생성
- 깨진 위키 링크 `[[...]]` 전수 스캔
- 빈 노트 / 거의 비어있는 노트 (200자 미만)
- 중복·유사 제목 후보 (자아·생각 폴더 우선)
- 결과: `위키/_정리노트/2026-05-04_점검리포트.md`
- **커밋**: `정리: 점검 리포트 생성`

### 2단계 — 위생 수정
- 깨진 링크 수정 (대상이 명확한 경우만; 모호하면 리포트에만 남김)
- 빈 노트 정리 (TODO만 있는 건 삭제, 의도 있어 보이면 보존)
- **커밋**: `정리: 위생 수정 (깨진 링크/빈 노트)`

### 3단계 — frontmatter 표준화
모든 노트에 다음 형식 통일:
```yaml
---
title: 노트 원래 제목
tags: [폴더기반_태그]
요약: 한 줄로 이 노트가 뭔지 (AI 검색 핵심)
생성일: YYYY-MM-DD
수정일: YYYY-MM-DD
---
```
- 기존 frontmatter 있으면 통합, 없으면 추가
- 태그는 폴더 경로 기반 자동 추론 (예: `위키/자아/` → `자아`)
- **커밋**: `정리: frontmatter 표준화`

### 4단계 — 자아·생각 폴더 중복 통합 (균형 기준)
- 같은 주제 + 70% 이상 겹칠 때만 통합
- 통합 시: 새 노트 생성 → 원본을 `위키/_아카이브/`로 이동 → 위키 링크 자동 갱신
- 커밋 메시지에 **왜 합쳤는지** 명시
- **커밋**: `정리: 자아·생각 중복 통합`

### 5단계 — MOC 신설 (자아 폴더 메타 인덱스)
- 파일: `위키/자아/_나는누구인가.md`
- 내용:
  - 자아 폴더 전체 노트 링크 (구조화)
  - 핵심 정체성·가치관·성향 한 페이지 요약
  - "이 사람을 한마디로" 첫 문단
- **커밋**: `정리: 자아 MOC '_나는누구인가' 신설`

### 6단계 — AI 페르소나 프롬프트 노트 신설
- 파일: `위키/자아/_AI에게_나를_소개하기.md`
- 내용: Claude Project 지침 또는 시스템 프롬프트에 그대로 붙일 수 있는 형식
  - 사용자 정체성 (1~2문단)
  - 사고방식·말투 특징
  - 답할 때 따라야 할 규범 ("위키 인용 시 출처 표시", "비개발자에게 한국어로" 등)
  - 자아 노트 참조 우선순위
- **커밋**: `정리: AI 페르소나 프롬프트 노트 신설`

### 7단계 — 색인 갱신
- `색인.md` 업데이트
  - 새 MOC, 페르소나 노트, 아카이브 폴더 반영
  - 폴더 구조 한눈에 보이게
- **커밋**: `정리: 색인 갱신`

## 사용자 검토 흐름 (작업 완료 후)

1. 7단계 끝나면 `cleanup/2026-05-04` 브랜치를 GitHub로 push
2. 사용자가 GitHub에서 main 대비 변경사항 확인 (또는 PR 생성)
3. 마음에 들면 → main과 머지
4. 마음에 안 들면 → 브랜치만 폐기, main은 그대로
